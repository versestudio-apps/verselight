→ Single source of truth = `pubspec.yaml`. Flutter tool writes the matching

MVP), so no `Info.plist` to keep in sync.

### 1.2 What has already used `versionCode=1`

| Phase | Artifact                                                  | Where it went |
|-------|-----------------------------------------------------------|---------------|
| 09D   | `...beta-android.apk` (debug-signed, vc=1)                | Internal sideload only, debug cert |

→ `vc=1` has been distributed to internal testers via sideload twice (debug

Amazon Appstore + Google Play both still see "this package does not exist".

### 1.3 Constraints / facts that shape the strategy

- Android `versionCode` is a 32-bit integer; max safe value `2 100 000 000`
  (Play Store cap; Amazon does not publish a cap but matches Play in practice).
- `versionCode` MUST strictly increase between two builds with the same
  `applicationId` if they should be upgradable on the same device — Android
  refuses installs that lower or repeat the existing `versionCode`.
- `versionName` is a free-form string shown to users; not used for ordering.
- Amazon Appstore rejects re-uploads with a `versionCode` already used in the
  same listing (same constraint as Play). Internal sideloads do **not** count
  against the store's bookkeeping — only what is uploaded to the developer
  console matters for the store side.
- Phases 09H–09L changed user-visible content (paraphrased verses, gated IAP,
  gated Audio, INTERNET permission added). The next signed APK is **not**
  identical to the Phase 09G artifact — it must therefore receive a higher
  `versionCode` even if it's only re-circulated as a beta.

## 2. Decision

### 2.1 Strategy: monotonic +1 `versionCode`, semantic `versionName`

```
versionName = SemVer (MAJOR.MINOR.PATCH) tied to user-facing change scope.
versionCode = monotonically increasing integer, +1 per new signed artifact
              published anywhere (beta sideload OR store upload).
```

**Concrete reservation plan:**

| versionCode | versionName | Purpose                                                  |
|-------------|-------------|----------------------------------------------------------|
| 1           | 1.0.0       | ✅ used — Phase 09G internal signed beta                  |
| 2–9         | 1.0.0       | Reserved for any new internal RC builds before submission |
| 10          | 1.0.0       | **First Amazon Appstore production submission**           |
| 11+         | 1.0.0       | Amazon resubmissions of 1.0.0 (rejection fixes, etc.)     |
| 20          | 1.0.1       | First patch release                                      |
| 30          | 1.1.0       | First minor release                                      |
| 100         | 2.0.0       | Hypothetical major                                       |

The "×10 between marketing versions" gap is a soft convention so a hotfix
in the same SemVer line never collides with the next marketing version's
seed. It's not load-bearing — Android only enforces monotonic increase. If
the gap fills up, just keep counting.

### 2.2 Why not date-based (`YYYYMMDDNN`)?

Considered and rejected for this app:

- Date-based schemes (`20260523_01`) work, but waste mental overhead for a
  solo-dev app that ships <1 build/day. Each new build forces a NN suffix
  computation and a sanity-check "is today's date past yesterday's?".
- Mapping `versionCode` → semantic intent (RC vs. production) is clearer
  with the reserved-band approach above than with a date.
- Rolling back is uglier with dates (can't pick `20260522_05` after
  `20260523_01` is already on the store — would have to leap forward).
- Monotonic +1 follows the KISS principle in
  [CLAUDE.md → Core Coding Principles](../../../README.md).

Date-based remains a valid fallback if the project ever moves to a CI/CD
release train shipping daily — revisit then.

### 2.3 Why versionName stays `1.0.0` for first submission

- First store release deserves `1.0.0` (signals "this is the v1, not a
  pre-release fragment").
- Phases 09H–09L improved content + safety without adding new user-facing
  features. SemVer `1.0.0` is appropriate; bumping to `1.0.1` for first
  submission would imply a patch on top of a never-published `1.0.0`,
  which is confusing.
- Patch (`1.0.1`) and minor (`1.1.0`) bumps wait until after the first
  release ships and feedback drives changes.

### 2.4 Why we are NOT bumping pubspec in THIS phase

This phase is intentionally **docs-only**, matching the pattern of Phases
09K (store listing audit) and 09L (Bible license review):

- The bump is mechanical and trivial (a single character in `pubspec.yaml`).
- Coupling the version bump to the actual submission commit gives cleaner
  git attribution — that commit is the canonical "this is what shipped to
  Amazon" marker.
- The repo HEAD producing a `vc=1` APK is not harmful as long as that APK
  never leaves the developer machine (it's only a smoke-test build).
- The recommendation is documented + linked from `KNOWN_ISSUES.md #8` so
  it cannot be silently forgotten at submission time.

If a fresh signed RC beta is needed before final submission, bump pubspec
to `1.0.0+2` in a one-line commit at that time. The first Amazon submission
commit bumps to `1.0.0+10`.

### 2.5 Exact one-line change to apply at submission time

In [pubspec.yaml:5](../../../pubspec.yaml):

```diff
-version: 1.0.0+1
+version: 1.0.0+10
```

Then `flutter clean && flutter build apk --release` to regenerate the APK
with `versionCode=10`. No Gradle or manifest change required.
